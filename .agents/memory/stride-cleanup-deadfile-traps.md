---
name: Stride cleanup dead-file traps
description: Files that LOOK like dead scaffolding/build output but are load-bearing — never delete during a cleanup audit
---

When running a "delete dead files" cleanup (e.g. a consultant audit doc), two items
look deletable but are load-bearing in the Replit-hosted + deployed setup. Do NOT delete:

- **`.replit-artifact/artifact.toml`** (every artifact has one) — defines the artifact's
  routing, ports, BASE_PATH, and the production `build`/`run`/`serve` commands and deploy
  config. Deleting it breaks workflows and the published app. An audit claiming "no relevance
  outside Replit" is misleading: the app runs and is published ON Replit.
- **`artifacts/stride-app/web-dist/`** — the exported web bundle. Generated by `pnpm run build`
  (`expo export`) and served in production by `server/web-proxy.js` + `server/static-serve.js`.
  It is build output, so the right move is `.gitignore` it (already done), not delete it.

**Why:** both pattern-match as "scaffolding" / "committed build output" that cleanup docs flag,
but removing them breaks routing/serving/deploy.
**How to apply:** during any cleanup batch, treat `.replit-artifact/*` and `web-dist/` as
protected; verify references (rg) before deleting anything, and prefer `.gitignore` over deletion
for regenerable build output. Safe-to-delete leftovers seen: orphan `app/(tabs)/` Expo group,
0-byte root `./src/services/emailService.ts`, empty plural `api-server/src/middlewares/` (keep
singular `middleware/`).
